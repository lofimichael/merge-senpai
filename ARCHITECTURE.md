# Merge Senpai - Architecture, Research Findings, and Hackathon Build Plan

Last updated: 2026-07-08

## 0. Executive Decision

2026-07-08 update: the hackathon install path is now a BYOK workflow installer,
not a GitHub App. The GitHub App architecture remains a reasonable future path
for first-class bot identity, but the current 80/20 is a single transparent
workflow copied into the user's repository by `install.sh`. This preserves the
"no Merge Senpai server sees your code" story, keeps all OpenAI usage on the
repository owner's key, and makes the install PR itself reviewable.

The workflow-mode tradeoff is identity: reviews and comments are authored by
`github-actions[bot]`. Merge Senpai branding lives in the workflow name, check
name, review body, fix commit author, labels, artifacts, media branch, and
installed avatar. A real `merge-senpai[bot]` identity still requires a GitHub
App or an optional machine-user token.

Merge Senpai should ship as a zero-server GitHub Action that uses a dedicated
OpenAI API key supplied by the installing repository. The minimum impressive
hackathon vehicle is:

1. Codex reviews a pull request in GitHub Actions and returns strict JSON.
2. Merge Senpai posts one GitHub PR review with a scorecard and inline
   suggestions.
3. The workflow job exits non-zero on blocking findings, so branch protection
   turns the PR red.
4. A maintainer comments `Senpai, fix this` from GitHub Mobile using OS
   dictation.
5. Codex applies a patch in the runner, commits it, and triggers re-review.
6. Supertonic synthesizes the in-character briefing locally.
7. `ffmpeg` composites a short H.264/AAC MP4 from static character art,
   subtitles, grade/status overlays, and the Supertonic audio.

This is the right 80/20 because it is operationally useful even without media:
review, gate, fix, and re-review work in public and private repos. The media
rail is a demo accelerator, not the dependency that makes the product work.

The optional "premium visual narrative" is OpenAI Images, not Higgsfield:
Codex writes a safe `image_prompt` in the same review JSON, then a script calls
OpenAI's Image API with the same `MERGE_SENPAI_OPENAI_KEY` to create one
non-code narrative still. `ffmpeg` then composites that still with the same
Supertonic audio. This should be off by default for cost, latency, org
verification, and privacy reasons.

The important correction from the prior document: do not make audio-only the
headline. The public demo should show a playable video briefing. For private
repos, never auto-publish media through public Pages or public raw links; use
text, job summary, and private workflow artifacts as the default private path.

## 1. Product Definition

One-liner:

> Merge Senpai is a code reviewer you can hear and talk back to. It reviews,
> blocks, explains, fixes, and re-reviews inside your CI, on your OpenAI key.

The closed loop:

```text
pull_request opened/synchronized
  -> Merge Senpai runs in the user's GitHub Actions runner
  -> Codex produces strict review JSON
  -> one GitHub PR review is posted
  -> job fails on P0/P1 findings, becoming the required merge gate
  -> optional media briefing is generated by Supertonic + ffmpeg
  -> maintainer comments "Senpai, fix ..."
  -> issue_comment workflow verifies authorization and same-repo branch
  -> Codex edits in workspace-write mode
  -> workflow commits/pushes the fix
  -> review workflow runs again
  -> red check flips green when blockers are gone
```

What is genuinely new:

- Voice/video is not the product by itself. The product is a closed loop that
  combines review, fix, and merge gate in the user's CI.
- The user does not onboard to SaaS. They add workflow YAML and a repository
  secret.
- The personality is repo-configurable without weakening the review contract.
- The "talk back from phone" loop uses GitHub Mobile plus OS dictation instead
  of building a separate mobile app.

## 2. Validated Research Findings

### 2.1 OpenAI and Codex

- The current Codex manual recommends `gpt-5.5` for strongest coding work and
  `gpt-5.4-mini` for faster/lower-cost lighter work. The older
  `gpt-5.3-codex` name in the previous architecture is now explicitly risky:
  the manual says `gpt-5.3-codex` is deprecated for ChatGPT sign-in, and API
  availability should be probed before pinning it.
- `codex exec` is the correct non-interactive primitive for CI automation.
  It supports JSONL, `--output-schema`, `--ephemeral`, sandbox modes, profiles,
  and `CODEX_HOME` configuration.
- The official `openai/codex-action@v1` is more capable than the earlier
  research assumed: it supports `codex-home`, `codex-args`, `sandbox`,
  `output-file`, model/effort inputs, and a key proxy/safety strategy. That
  means we should not claim "stock action exposes none of this" anymore.
- For the hackathon, either direct CLI or `openai/codex-action` can work. The
  safer default is to use the official action for Codex invocations if
  `codex-args` can carry `--output-schema` and persona profile args cleanly.
  Direct CLI remains acceptable if we scope the API key to the single `codex`
  process and configure environment exclusion.

### 2.2 OpenAI Images

- `gpt-image-2` is available through the Image API and can generate images from
  text prompts.
- The Image API and Responses API both support image generation. The Image API
  lets us choose `gpt-image-2` directly; the Responses API adds mainline model
  token costs in addition to image costs, so the Image API is the better fit for
  the optional visual narrative.
- OpenAI notes that GPT Image models may require API Organization Verification.
  This is a real hackathon risk and must be probed before relying on it.
- Image generation pricing is token-based. As of the 2026-07-08 docs,
  `gpt-image-2` pricing is listed at $8 / $2 / $30 per 1M image input /
  cached input / output tokens, and $5 / $1.25 per 1M text input / cached input
  tokens. We should use OpenAI's image calculator or the returned usage data for
  real per-image costs rather than promising a fixed number.
- OpenAI business/API data is not used for model training by default unless the
  organization opts in. Abuse-monitoring retention can still apply. Our copy
  must say: "code and prompts go to OpenAI on your key," not "code never leaves."

### 2.3 Supertonic

- Supertonic is a strong default for the minimum media rail. The upstream repo
  describes it as ONNX Runtime based, local/on-device, no cloud/API calls, 31
  languages, about 99M parameters, no GPU requirement, 44.1 kHz WAV output, and
  expression tags such as `<laugh>`, `<breath>`, and `<sigh>`.
- This keeps media generation cheap and privacy-aligned: review-specific text
  does not go to a second non-OpenAI vendor.
- We should cache the model directory in GitHub Actions. First run may pay the
  model download; demo should pre-warm it.

### 2.4 ffmpeg and GitHub Runners

- Do not assume `ffmpeg` is preinstalled on `ubuntu-latest`. The current
  Ubuntu 24.04 runner image docs enumerate installed software and do not list
  ffmpeg, even though `MediaInfo` is present. The action should run:

```bash
if ! command -v ffmpeg >/dev/null 2>&1; then
  sudo apt-get update
  sudo apt-get install -y ffmpeg
fi
```

- Install `ffmpeg` before any `drop-sudo` or unprivileged Codex step.
- Encode video as conservative mobile-compatible MP4:
  H.264 video, AAC audio, `yuv420p`, `+faststart`.

### 2.5 GitHub Reviews, Comments, Tokens, and Media

- The REST "Create a review for a pull request" endpoint supports one review
  body plus inline comments, and review events include `APPROVE`,
  `REQUEST_CHANGES`, and `COMMENT`.
- Review comments need valid diff positions/lines. One bad inline anchor can
  cause a 422, so the hunk filter is a required component, not polish.
- The issue comment REST endpoint accepts a `body` string. It does not expose a
  documented file-upload/attachment parameter. Treat dynamic comment attachment
  uploads as unavailable.
- `GITHUB_TOKEN` pushes do not normally trigger new workflow runs. GitHub's
  documented exceptions include `workflow_dispatch` and `repository_dispatch`,
  so the zero-setup fallback is: after a fix commit pushed by `GITHUB_TOKEN`,
  explicitly dispatch the review workflow.
- A GitHub App installation token or PAT can push under a better identity and
  avoid some `GITHUB_TOKEN` limitations. It is optional for installs and useful
  for the demo.
- GitHub Actions artifacts are a good private-repo media fallback. GitHub added
  non-zipped artifact support in 2026; with `archive: false`, supported file
  types can be viewed directly in the browser. This is private/authenticated and
  does not require public Pages.
- Public GitHub Pages are fine for the demo player. Private Pages access control
  exists for Enterprise Cloud, but ordinary private repos cannot rely on it.
  Never enable a user's Pages site to publish private PR media.

### 2.6 Media Delivery Reality

From the prior live probes and attached screenshots:

- User-uploaded `.mov`/`.webm` attachments render inline in GitHub comments.
  Dynamic bot uploads are not supported by documented APIs.
- A committed public raw `.mp4` URL is served by `raw.githubusercontent.com` as
  `application/octet-stream`; direct taps can download instead of playing.
- A static HTML player with `<video src="raw-url">` can stream the same raw MP4
  in a browser. That makes a public player page viable.
- The GitHub mobile blob page for a committed `test.mp4` did not show a native
  preview in the attached screenshot; it showed file metadata and "View raw."
  Do not rely on blob preview as the private/mobile delivery mechanism.

## 3. Minimum Delivery Vehicle

The product should be installable as open-source YAML plus one reusable action.

### User-facing install

```yaml
name: Merge Senpai

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  contents: read
  pull-requests: write
  issues: write
  actions: write

jobs:
  senpai:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
          persist-credentials: false

      - uses: merge-senpai/merge-senpai@v1
        with:
          openai-api-key: ${{ secrets.MERGE_SENPAI_OPENAI_KEY }}
          model: gpt-5.4-mini
          media: auto
          visual-mode: static
```

`MERGE_SENPAI_OPENAI_KEY` is intentional. Do not call the secret
`OPENAI_API_KEY` in examples, because many repos already use that for
production/ops. The action maps it internally to the minimum env needed by the
specific Codex/OpenAI step.

### Recommended secret split

- `MERGE_SENPAI_OPENAI_KEY`: OpenAI project key for this reviewer only.
- Existing app/ops keys: untouched, never referenced.
- Optional demo identity:
  - `MERGE_SENPAI_APP_ID`
  - `MERGE_SENPAI_APP_PRIVATE_KEY`
  - or a fine-grained machine-user PAT if the App path fails.

### Why this works in any repo

The core product works on public and private repos:

- Review text posts to the PR.
- Required check gates merges.
- Maintainer-authenticated comment commands can trigger fixes on same-repo
  branches.
- Private code/media does not pass through Merge Senpai servers because there
  are no Merge Senpai servers.

Media behavior differs by repo visibility:

| Repo type | Default behavior | Why |
|---|---|---|
| Public | Generate MP4, commit to `senpai-media` orphan branch, link through static player page | Best demo/user experience; no server; public raw URL is acceptable for public code |
| Private | Generate text review and optional private workflow artifact; do not link a public player | Keeps privacy story clean; browser raw auth and mobile preview are not reliable |
| Enterprise with private Pages | Optional private player if the org already supports Pages access control | Not a general install assumption |

## 4. System Architecture

```text
Installing repository
  .github/workflows/senpai-review.yml
  .github/workflows/senpai-fix.yml
  .github/senpai.yml
  secrets.MERGE_SENPAI_OPENAI_KEY
  optional GitHub App/PAT identity secrets

merge-senpai/merge-senpai action
  action.yml
  codex-home/
    config.toml
    tsundere.config.toml
    kouhai.config.toml
  schema/review.schema.json
  prompts/review.md
  assets/
    tsundere.png
    kouhai.png
  scripts/
    resolve-config.mjs
    run-codex-review.sh
    post-review.mjs
    synthesize-supertonic.py
    maybe-generate-image.mjs
    compose-video.sh
    dispatch-review.mjs
    codex-fix.sh
  player/index.html

GitHub surfaces
  PR review body and inline comments
  required status check
  workflow summary
  optional Actions artifact
  optional public media branch
  optional static public player page
```

Data flows:

1. Repo context/diff -> Codex/OpenAI API using the user's key.
2. Optional safe image prompt -> OpenAI Image API using the same dedicated key.
3. Briefing script -> Supertonic locally in runner.
4. Audio/image/static art -> `ffmpeg` locally in runner.
5. Review JSON/media paths -> GitHub API in the user's repository.

No data flows to Merge Senpai infrastructure. There is no webhook, database,
queue, proxy, hosted renderer, or telemetry service.

## 5. Review JSON Contract

Codex must produce a strict JSON object. Persona can affect tone fields only.

```json
{
  "verdict": "request_changes",
  "grade": "C-",
  "summary": "Short human-readable summary.",
  "persona_line": "I-it's not like I wanted this to pass, but...",
  "briefing_script": "A 60-90 second spoken script, no secrets.",
  "image_prompt": "Optional safe visual prompt with no code snippets.",
  "findings": [
    {
      "severity": "P0",
      "title": "SQL injection in login lookup",
      "file": "src/db.ts",
      "line": 42,
      "message": "Clinical explanation, no persona fluff.",
      "suggestion": "parameterized replacement code if safe"
    }
  ]
}
```

Rules:

- `findings[].message`, severity, and suggestions stay clinical.
- Persona applies only to `persona_line`, `summary`, `briefing_script`, and
  optionally `title`.
- `briefing_script` must summarize the verdict, finding count, and one next
  action. It must not read long code, secrets, env vars, tokens, or private URLs.
- `image_prompt` must be abstract and safe: no code snippets, no file contents,
  no secrets, no customer names unless the repo config explicitly permits it.

## 6. Codex Execution Strategy

### Preferred implementation path

First probe whether `openai/codex-action@v1` can carry the exact args we need:

- `codex-home` set to our shipped persona config.
- `codex-args` includes `--output-schema`, `--ephemeral`, `--profile`, and
  per-run `-c developer_instructions=...`.
- `output-file` writes strict JSON.
- `sandbox` is `read-only` for review and `workspace-write` for fix.
- `safety-strategy` is left on the secure default where possible.

If that works, use the official action internally or document a starter
workflow that composes it with our posting/media scripts. This gives us the
OpenAI key proxy and security posture for free.

If it does not work, use direct CLI:

```bash
export CODEX_HOME="$RUNNER_TEMP/merge-senpai-codex-home"
CODEX_API_KEY="$INPUT_OPENAI_API_KEY" codex exec \
  --model "$INPUT_MODEL" \
  --profile "$PERSONA" \
  --strict-config \
  --sandbox read-only \
  --output-schema "$ACTION_PATH/schema/review.schema.json" \
  -o "$RUNNER_TEMP/review.json" \
  --ephemeral \
  -C "$GITHUB_WORKSPACE" \
  -
```

Direct CLI rules:

- Never set `OPENAI_API_KEY`, `CODEX_API_KEY`, `GITHUB_TOKEN`, or app private
  keys at job scope.
- Pass the OpenAI key only to the Codex process or Image API process that needs
  it.
- Configure `shell_environment_policy` to exclude `*KEY*`, `*TOKEN*`,
  `*SECRET*`, and app-private-key env vars from Codex's tool shell.
- Run repo setup/test/build steps before exposing the OpenAI key.
- Use `read-only` sandbox for review and `workspace-write` only after an
  authorized maintainer comment.

### Model default

For the hackathon:

- Default to `gpt-5.4-mini` for cost and speed.
- Demo can switch to `gpt-5.5` if the first real-key probe shows material review
  quality improvement and runtime remains acceptable.
- Do not pin `gpt-5.3-codex` unless the key probe proves availability and the
  team accepts the deprecation risk.

## 7. Persona System

V1 ships two personas:

- `tsundere`: default demo persona, stern but caring.
- `kouhai`: first-time contributor mode, warmer and more explanatory.

Config:

```yaml
version: 1
persona: tsundere
persona_level: 2
block_on: [P0, P1]
media: auto             # auto | off | video
visual_mode: static     # static | openai-image
image_quality: low      # low | medium | high, passed through only if supported
private_media: artifact # artifact | off
```

Precedence:

1. Built-in defaults.
2. Action inputs.
3. `.github/senpai.yml` read from the base ref.
4. First-time contributor override to `kouhai`.
5. Future: author map and PR labels.

Security:

- Read `.github/senpai.yml` from the base branch, not from the PR head.
- Validate persona names. Unknown profile names must fail closed because some
  Codex versions silently fall back to vanilla config.

## 8. Posting and Gate

`post-review.mjs` responsibilities:

1. Parse `git diff --unified=0 origin/$BASE...HEAD`.
2. Build a valid line/position map for each changed file.
3. Keep only inline findings that can be anchored to the diff.
4. Fold unanchorable findings into the body.
5. POST one PR review:
   - `event: REQUEST_CHANGES` if blockers exist.
   - `event: APPROVE` if clean and approval is permitted.
   - fallback to `COMMENT` if approval/request-changes is blocked.
6. Include a media block only after media generation succeeds.
7. Write the same scorecard to `$GITHUB_STEP_SUMMARY`.
8. Exit 1 only after posting if any severity is in `block_on`.

The status check is the merge gate. Branch protection requires the `senpai`
workflow/job.

## 9. Fix Loop

The fix loop lives in a separate workflow on the default branch:

```yaml
on:
  issue_comment:
    types: [created]

jobs:
  fix:
    if: >-
      github.event.issue.pull_request != null &&
      (
        startsWith(github.event.comment.body, '/senpai') ||
        startsWith(github.event.comment.body, 'Senpai,')
      ) &&
      contains(fromJSON('["OWNER","MEMBER","COLLABORATOR"]'),
               github.event.comment.author_association)
```

Steps:

1. React to the comment so the user sees acknowledgement.
2. Resolve PR head repo/branch.
3. Hard-fail if head repo != base repo for v1.
4. Checkout PR branch with write token.
5. Fetch previous review JSON/body.
6. Run Codex in `workspace-write` mode with the user's comment and prior review.
7. Commit only if the diff is non-empty.
8. Push.
9. Trigger re-review:
   - If using App/PAT, push should trigger `pull_request:synchronize`.
   - If using `GITHUB_TOKEN`, dispatch the review workflow explicitly.

The dictation grammar is natural language:

- `/senpai fix the SQL injection and add a regression test`
- `Senpai, fix the SQL injection and add a regression test`

## 10. Media Rail

### 10.1 Minimum media: static art + Supertonic + ffmpeg

This is the hackathon default.

Inputs:

- `assets/tsundere.png` or `assets/kouhai.png`
- `review.json.briefing_script`
- `review.json.grade`
- finding count and blocker count

Pipeline:

```text
briefing_script
  -> Supertonic WAV
  -> generated SRT subtitle chunks
  -> ffmpeg composite:
       static character art
       subtle zoom/pan or looped idle background
       grade badge overlay
       blocker count overlay
       burned-in subtitles
       AAC audio
  -> senpai-briefing.mp4
```

Example ffmpeg shape:

```bash
ffmpeg \
  -loop 1 -framerate 30 -i "$ART" \
  -i "$WAV" \
  -vf "scale=1280:720:force_original_aspect_ratio=increase,crop=1280:720,subtitles=$SRT" \
  -c:v libx264 -pix_fmt yuv420p -preset veryfast -movflags +faststart \
  -c:a aac -b:a 128k -shortest "$OUT_MP4"
```

Fallbacks:

- If Supertonic fails: skip media and keep the gate.
- If ffmpeg install fails: upload WAV/MP3 artifact or skip media.
- If `libx264` is unavailable: retry with an installed codec, but keep MP4/H.264
  as the target for demo reliability.

### 10.2 Optional OpenAI image narrative

This is a feature flag, not the minimum.

Flow:

1. Codex produces `image_prompt` in strict JSON.
2. `maybe-generate-image.mjs` sanitizes it:
   - remove code blocks
   - remove URLs
   - remove secrets-like strings
   - cap length
   - force "no readable source code or secret text"
3. Call the Image API:

```bash
curl -sS https://api.openai.com/v1/images/generations \
  -H "Authorization: Bearer $MERGE_SENPAI_OPENAI_KEY" \
  -H "Content-Type: application/json" \
  -d @image-request.json \
  | jq -r '.data[0].b64_json' \
  | base64 --decode > narrative.png
```

4. Use `narrative.png` as the ffmpeg visual input.

Rules:

- `visual_mode: static` is default.
- `visual_mode: openai-image` is opt-in.
- Generate at most one image per PR review.
- Add a repo-level budget switch:
  `max_visual_generations_per_pr: 1`.
- If org verification blocks the model, fall back to static art.
- The image prompt must not include proprietary code or secrets. For private
  repos, default this off unless explicitly enabled.

Why optional:

- It uses the same OpenAI key, so no second vendor is added.
- It can create a much more memorable "status report image narrative."
- It adds latency, cost, and verification risk.
- It sends an additional prompt to OpenAI, so private repos should opt in.

### 10.3 OpenAI TTS fallback

Supertonic remains the default. If venue audio quality is poor, the same OpenAI
key can call `gpt-4o-mini-tts`. OpenAI's current TTS docs show an
`audio/speech` endpoint with `model: gpt-4o-mini-tts`, `voice`, `input`, and
`instructions`, returning MP3 by default.

Use this only as `voice: premium` or demo fallback because it adds cost and
sends the spoken script to OpenAI. It does not add a new vendor.

## 11. Media Delivery

### Public repos and demo path

Public path:

1. Commit `senpai-briefing.mp4` to orphan branch `senpai-media`.
2. Use a unique filename:
   `pr-<number>-<short-sha>-<run-id>.mp4`.
3. Post a PR review link to:
   `https://merge-senpai.github.io/play#v=<raw-mp4-url>`.
4. Static player reads only the URL fragment, so the media URL is not sent to
   the Pages server in normal HTTP requests.
5. Also include a link to the workflow artifact as backup.

The player page is not a data service. It is a static HTML/JS file with no
analytics, no cookies, and no API calls except fetching the media URL the user
already has.

### Private repos

Private path:

1. Always post the text review and gate.
2. If media generation is enabled, upload `senpai-briefing.mp4` as a private
   workflow artifact with short retention.
3. Add the artifact/run link to the PR review and job summary.
4. Optionally commit to a private `senpai-media` orphan branch and link the
   GitHub blob page, but mark it "best effort" because mobile playback is not
   reliable.
5. Never route private media through public `merge-senpai.github.io`.
6. Never enable GitHub Pages in the user's private repo automatically.

This means private installs "work" fully, but the wow media is less seamless.
That is the right tradeoff: privacy and installability beat forcing a leaky or
fragile player.

### User-attachments policy

Use `github.com/user-attachments/assets/<uuid>` only for pre-uploaded static
brand stingers. Never put repo-derived content in user attachments. Never use
browser session cookies or undocumented upload endpoints to create dynamic
attachments.

## 12. Identity and Tokens

The GitHub App is optional.

| Strategy | Setup | Identity | Fix retrigger |
|---|---|---|---|
| T1: plain `GITHUB_TOKEN` | zero extra setup | `github-actions[bot]` | use `workflow_dispatch` fallback |
| T2: fine-grained PAT | one extra secret | machine user avatar | push generally retriggers |
| T3: GitHub App installation token | app registration + private key secret | `merge-senpai[bot]` | best demo/prod identity |

Recommended:

- Public README defaults to T1.
- Demo uses T3 if the first probe confirms push/retrigger behavior.
- Keep T2 PAT as a backup for the hackathon.

No webhook is required for the app-token path. The app is an identity and
installation-token mechanism, not a server integration.

## 13. Privacy and Security Posture

What we can honestly say:

- "No Merge Senpai server sees your code, key, media, or review."
- "The action runs in your GitHub Actions runner."
- "State lives in your repo, PR, workflow run, or artifact storage."
- "OpenAI receives the prompt/repo context needed by Codex on your key."
- "OpenAI Images is optional and off by default for private repos."
- "Supertonic and ffmpeg run locally in CI."

What we must not say:

- "Your code never leaves." Codex sends code context to OpenAI.
- "Private media is one-tap playable everywhere." It is not.
- "No third party receives anything." OpenAI receives Codex prompts and
  optional image/TTS prompts on the user's key.

Hard requirements:

- Read user config from the base ref.
- Never use `pull_request_target` for agentic review/fix of PR code.
- Same-repo branches only for fix mode in v1.
- Maintainer/collaborator author-association gate for fix commands.
- API key scoped to the specific step/process that needs it.
- No dynamic user-attachment uploads.
- No telemetry in the player.
- No public media publishing for private repos.

## 14. Verification Matrix

Run these before implementation is considered demo-ready.

| Probe | Purpose | Pass condition | Fallback |
|---|---|---|---|
| P1: Codex action arg probe | Verify `openai/codex-action@v1` can run with shipped `codex-home`, persona profile, `--output-schema`, and output file | Valid `review.json` from demo PR | Direct CLI with scoped `CODEX_API_KEY` |
| P2: model/key probe | Verify hackathon API key can use selected model | `gpt-5.4-mini` or `gpt-5.5` review succeeds | switch model input |
| P3: GitHub App retrigger | Verify App-token push retriggers review | fix commit produces new review run | `workflow_dispatch` fallback or PAT |
| P4: Supertonic CI | Verify pip install/model download/synthesis on ubuntu runner | WAV produced and cached | OpenAI TTS fallback or no media |
| P5: ffmpeg CI | Verify install and MP4 composition | H.264/AAC MP4 plays locally/browser | upload audio/text only |
| P6: public player | Verify raw MP4 loads through static `<video>` player on desktop and phone | video plays from PR link | artifact link backup |
| P7: private artifact | Verify private repo media artifact is accessible to collaborator and not public | signed-in collaborator can view/download; signed-out gets blocked | text-only private mode |
